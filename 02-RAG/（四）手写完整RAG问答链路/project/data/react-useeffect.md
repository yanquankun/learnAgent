## 为什么 useEffect 会执行两次

很多同学在开发环境发现 useEffect 执行了两次，第一反应是自己写出了 bug。
实际上这是 React 18 严格模式（StrictMode）的故意设计：开发环境下组件会被
故意挂载、卸载、再挂载一次，用来帮你提前暴露「副作用没有正确清理」的问题。
生产环境不会出现这个行为，所以不要为了「修复」它而去掉 StrictMode。

## 依赖数组的三种形态

useEffect 的第二个参数（依赖数组）有三种写法，语义完全不同：

不传依赖数组时，副作用在每次渲染后都会执行，这通常不是你想要的。
传空数组时，副作用只在组件挂载后执行一次，等价于 componentDidMount。
传具体依赖时，只有依赖项变化后才会重新执行副作用。

## 最容易踩的坑：对象和函数作为依赖

JavaScript 中对象和函数是引用类型，每次渲染都会创建新的引用。
如果把组件内定义的对象或函数直接放进依赖数组，每次渲染引用都变了，
副作用就会无限重复执行。

解决方案有三个：
第一，用 useMemo 缓存对象，保证引用稳定。
第二，用 useCallback 缓存函数。
第三，最推荐的方式是把依赖「降级」为原始值，比如只依赖 user.id 而不是整个 user 对象。

## 清理函数：被遗忘的重要细节

useEffect 可以返回一个清理函数，它会在组件卸载、以及下一次副作用执行之前被调用。
订阅事件、启动定时器、发起可取消的请求，都必须在清理函数里做对应的清理，
否则就会出现内存泄漏或者「组件已卸载还在 setState」的警告。

## 总结

依赖数组的本质是「告诉 React 副作用读取了哪些值」。写对依赖的关键是：
诚实地列出所有用到的响应式值，然后用 useMemo、useCallback 或依赖降级
来解决引用不稳定的问题，而不是欺骗 React 少写依赖。
